# Kamstrup N-ary Mapping Problem — Solution Proposals

**Date:** February 6, 2026  
**Prepared by:** Ole (analysis of OmniGaze codebase + Kamstrup data model)  
**Context:** How to represent KamstrupData's 5,396-row fact table (Application × BusinessCapability × Organization × Process × ValueStream with contextual TIME/Criticality) in OmniGaze's binary relationship model.

---

## The Problem in Detail

Each KamstrupData row is semantically: *"Application X supports Capability Y for Organization Z via Process P in Value Stream V — and in THIS specific context, its TIME rating is T and Business Criticality is C."*

OmniGaze stores relationships as `ParentFactSheetsIds` (a `List<Guid>` on each FactSheet). This is fundamentally **binary** — factsheet A references factsheet B. There is no native way to say "A relates to B *in the context of* C, D, and E, *with attributes* TIME=Invest, Criticality=MissionCritical."

### What we need to preserve:
1. **The n-ary combination** — which 5 entities go together in a single mapping row
2. **Contextual TIME** — the same app can be "Invest" in one capability but "Migrate" in another
3. **Contextual Business Criticality** — same deal, varies per context
4. **Lifecycle per mapping** — FromDate/EndDate on the mapping, not the app
5. **Additional context** — Comments, Projects, Value Stream Order

---

## Solution 1: Flatten to Binary Relationships + Contextual Concerns (Minimal Dev)

### How It Works
Import all binary relationships from KamstrupData:
- Application → BusinessCapability (via `ParentFactSheetsIds`)
- Application → UserGroup/Organization (via `ParentFactSheetsIds`)
- Application → Process (via `ParentFactSheetsIds`)
- BusinessCapability → ValueStream (via `SupportedByCapabilityIds`)

For contextual attributes, create **FactSheetConcerns on the Application** that encode the context:

```
Concern.Header = "Capability Context: Strategic Management"
Concern.Description = "TIME: Invest | Criticality: MissionCritical | Org: DK_Digitalization | Process: P10-Sourcing"
Concern.ConcernCriticality = High (mapped from BusinessCriticality)
Concern.AutoGenerated = true
```

### What Exists Today
- ✅ `ParentFactSheetsIds` — fully working bidirectional relationship system
- ✅ `FactSheetConcern` — has Header, Description, ConcernCriticality, RiskHandling, AutoGenerated
- ✅ `RelationshipManager` — knows all valid relationship types between factsheet types
- ✅ All binary relationship paths already defined in `RelationshipManager.InitializeRelationshipRules()`

### What Needs Dev
- Import script to decompose 5,396 n-ary rows into distinct binary relationships (~2h)
- Concern-generation logic that encodes context into structured Description fields (~2h)
- A parser/query layer that can reconstruct the n-ary view from concerns (~4h if needed)

### Pros
- **Zero model changes** — works with today's OmniGaze, no protobuf schema changes
- **Fast to implement** — can be done in a day
- **All existing UI works** — capability maps, app grids, relationship views all work immediately
- **Concerns appear on the FactSheet detail page** — visible to users

### Cons
- **Context is lossy** — you can see that App X maps to Capability Y and Org Z *separately*, but can't easily answer "what is the TIME for App X *specifically in* Capability Y for Org Z?"
- **Concern descriptions are unstructured text** — querying them requires string parsing
- **Concern count explosion** — an app with 50 capability mappings gets 50 auto-generated concerns, cluttering the UI
- **No clean reporting** — can't build a grid/pivot table from concern text

### Verdict: ⚠️ Good for v1 quick win, but insufficient for Kamstrup's analytical needs

## Solution 2: ApplicationContextMapping FactSheet Type (Recommended)

### How It Works
Create a new **`ApplicationContextMapping`** factsheet type that explicitly models one row of KamstrupData. Each mapping instance IS a factsheet, carrying:

```csharp
[ProtoContract]
public class ApplicationContextMappingFactSheet : FactSheet
{
    // The core n-ary reference (all as Guids referencing other factsheets)
    [ProtoMember(1)] public Guid ApplicationId { get; set; }
    [ProtoMember(2)] public Guid BusinessCapabilityId { get; set; }
    [ProtoMember(3)] public Guid OrganizationId { get; set; }      // UserGroupFactSheet
    [ProtoMember(4)] public Guid ProcessId { get; set; }
    [ProtoMember(5)] public Guid ValueStreamId { get; set; }
    
    // Contextual attributes (the whole point)
    [ProtoMember(10)] public PortFolioStrategy.StrategyEnum ContextualTIME { get; set; }
    [ProtoMember(11)] public BusinessSupport.CriticalityEnum ContextualCriticality { get; set; }
    
    // Lifecycle of the mapping itself
    [ProtoMember(12)] public DateTime? MappingFromDate { get; set; }
    [ProtoMember(13)] public DateTime? MappingEndDate { get; set; }
    
    // Additional context
    [ProtoMember(14)] public string Comment { get; set; }
    [ProtoMember(15)] public int ValueStreamOrder { get; set; }
    [ProtoMember(16)] public List<Guid> ProjectIds { get; set; } = new();
}
```

The `DisplayName` could be auto-generated: "IFS ERP → Strategic Management (DK_Digitalization)"

The 5,396 KamstrupData rows become 5,396 ApplicationContextMapping factsheets. They participate in the standard ParentFactSheetsIds system too (referencing the App, Capability, Org, Process, ValueStream as parents), so existing navigation works.

### What Exists Today
- ✅ `FactSheet` base class — all infrastructure (ProtoContract, serialization, concerns, lifecycle, custom fields, owners, audit)
- ✅ `FactSheetContainer` — just needs a new `List<ApplicationContextMappingFactSheet>` added
- ✅ `RelationshipManager` — add new rules for this type
- ✅ `FactSheetMetaModel` — already supports extension with new types
- ✅ `CustomColumnDefinition` — works on any factsheet type, so custom fields for free
- ✅ All grid/search/export infrastructure works on any FactSheet subclass

### What Needs Dev
1. **New class** `ApplicationContextMappingFactSheet` — ~100 lines of C# (1h)
2. **FactSheetContainer update** — add `ProtoMember(19)`, `List<ApplicationContextMappingFactSheet>`, update `AllFactSheets`, `RemoveFactSheet` (30min)
3. **RelationshipManager rules** — add relationship rules (30min)
4. **Import script** — maps KamstrupData rows to these factsheets (4h)
5. **UI: List page** — standard grid for browsing mappings (2h with existing `StandardGrid2`)
6. **UI: Detail page** — show the 5 referenced entities + contextual attributes (2h)
7. **UI: Capability map integration** — when clicking an app in a capability, show its contextual TIME/criticality from these mappings instead of the global one (4h)
8. **Total: ~14h (2 days)**

### Pros
- **Full fidelity** — every KamstrupData row is preserved exactly, with all contextual attributes
- **Queryable** — `FactSheetContainer.ApplicationContextMappings.Where(x => x.ApplicationId == app.Id && x.ContextualTIME == Migrate)` — clean, typed queries
- **Standard FactSheet benefits** — gets custom fields, concerns, audit trail, search, export for free
- **Reversible** — if it doesn't work out, the binary relationships are ALSO maintained (the mapping just references them), so you can drop the mapping type without losing the basic relationships
- **Familiar pattern** — this is exactly what LeanIX calls "Application-to-Capability relation with relation properties" — a proven EA pattern
- **Scalable** — 5,396 factsheets is well within OmniGaze's capacity (existing customers have 10K+)

### Cons
- **5,396 additional factsheets** — increases the dataset, but not problematically
- **UI overhead** — needs at minimum a list page and contextual display (2-4h dev)
- **Conceptual complexity** — "a mapping as a factsheet" is a less intuitive concept for end users (mitigated by not exposing it prominently — it's a behind-the-scenes type)
- **Serialization growth** — 5K extra ProtoContract objects in the .bin file (negligible — each is tiny)

### Verdict: ✅ **Recommended approach.** Best fidelity, clean architecture, leverages existing infrastructure, reasonable dev effort.

## Solution 3: Relationship Properties via MetaModel Extension

### How It Works
Extend the MetaModel system to support **properties on relationships**, not just on factsheets. Today `FactSheetMetaModel` stores `Dictionary<Guid, string> FactSheetTextValues` keyed by FactSheet ID. Extend this to support a **composite key** — two FactSheet IDs forming a relationship:

```csharp
// New: relationship-level properties
[ProtoMember(12)] public Dictionary<string, string> RelationshipTextValues { get; set; } = new();
// Key format: "{sourceGuid}|{targetGuid}|{propertyName}"
// e.g., "{appGuid}|{capabilityGuid}|TIME" = "Invest"
```

Or more cleanly, create a new `RelationshipMetaModel` class:

```csharp
[ProtoContract]
public class RelationshipProperty
{
    [ProtoMember(1)] public Guid SourceFactSheetId { get; set; }
    [ProtoMember(2)] public Guid TargetFactSheetId { get; set; }
    [ProtoMember(3)] public string PropertyName { get; set; }  // "TIME", "Criticality"
    [ProtoMember(4)] public string PropertyValue { get; set; }  // "Invest", "MissionCritical"
    [ProtoMember(5)] public string Context { get; set; }         // Additional context (org, process)
}
```

### What Exists Today
- ✅ `FactSheetMetaModel` — has the pattern of `Dictionary<Guid, string>` for per-factsheet properties
- ✅ `FactSheetMetaModelContainer` — singleton load/save pattern
- ✅ `MetaModelConfiguration.razor.cs` — admin UI for configuring meta models

### What Needs Dev
1. **New `RelationshipProperty` class** (1h)
2. **Container for relationship properties** with efficient lookup (2h)
3. **UI for viewing/editing relationship properties** (4h)
4. **Import script** that creates relationship properties for TIME/Criticality on each app→capability pair (3h)
5. **Problem: this only gives you binary relationship properties.** The n-ary context (org, process) would need to be encoded as additional properties or in the Context field — effectively a structured string again.
6. **Total: ~10h, but with significant limitations**

### Pros
- **Extends existing MetaModel pattern** — conceptually consistent with how OmniGaze already works
- **Doesn't create new factsheet types** — less conceptual overhead
- **Works for the binary case** — if Kamstrup only needed per-app-per-capability attributes (no org/process context), this would be elegant

### Cons
- **Still fundamentally binary** — can express "App X's TIME for Capability Y is Invest" but NOT "App X's TIME for Capability Y *in Org Z via Process P* is Invest" without resorting to composite keys/context strings
- **No standard FactSheet benefits** — relationship properties don't get concerns, custom fields, search, export for free
- **Query complexity** — looking up properties requires custom Dictionary lookups instead of LINQ on typed objects
- **UI investment** — relationship properties need their own display layer (MetaModel config UI exists but doesn't cover relationships)
- **Scale concern** — 5,396 × 2+ properties = 10K+ dictionary entries in a single Dictionary — not terrible, but not as clean as typed objects

### Verdict: ⚠️ Elegant for simple binary relationship attributes, but insufficient for Kamstrup's full n-ary model. Could complement Solution 2 for simpler customers.

## Solution 4: Denormalize with Tags + Custom Fields

### How It Works
Import all binary relationships normally. Then for each Application, create structured **CustomField values** that encode the contextual attributes per capability:

Define custom columns:
- `CapabilityTIME_<CapabilityName>` (Dropdown: Tolerate/Invest/Migrate/Eliminate)
- `CapabilityCriticality_<CapabilityName>` (Dropdown: Administrative/Operational/Critical/MissionCritical)

Or more practically, create a single custom field per app:
- `ContextualMappings` (TextArea) — containing a JSON blob:

```json
[
  {
    "capability": "Strategic Management",
    "org": "DK_Digitalization", 
    "process": "P10-Sourcing",
    "time": "Invest",
    "criticality": "MissionCritical",
    "valueStream": "P2P"
  },
  ...
]
```

### What Exists Today
- ✅ `CustomColumnDefinition` — full-featured with 14 field types including TextArea, Dropdown, MultiSelect
- ✅ `CustomFieldValue` — stored as `List<CustomFieldValue>` on every FactSheet
- ✅ Complete admin UI for defining custom columns
- ✅ Grid integration, search, export all support custom fields
- ✅ Validation system built in

### What Needs Dev
1. Define custom column definitions (~1h via admin UI, or programmatic)
2. Import script to populate CustomField values (2h)
3. If using JSON blob approach: custom rendering component for the JSON (4h)
4. **Total: 3-7h depending on approach**

### Pros
- **Zero model changes** — uses existing CustomFields system completely
- **Visible in grids** — custom fields show up in StandardGrid2 automatically
- **Groupable/filterable/sortable** — all grid features work on custom fields
- **Admin-configurable** — non-developers can add/modify fields

### Cons
- **Column explosion** — if creating per-capability fields, you'd need ~673 × 2 = 1,346 custom columns per app (absurd)
- **JSON blob approach** — loses all query/filter/sort capabilities (it's just text)
- **Not relational** — can't navigate from the custom field to the actual Capability/Org/Process factsheets
- **Maintenance nightmare** — if a capability name changes, custom column names become stale
- **Doesn't solve the n-ary problem** — just pushes the complexity into unstructured text

### Verdict: ❌ Not recommended for the n-ary mapping. Fine for simple per-app attributes like "Crown Jewels" or "NIS2", but wrong tool for relational context.

## Solution 5: Virtual Computed View (Query-Time Aggregation)

### How It Works
Don't store the n-ary mapping at all. Instead:
1. Import all binary relationships (app→capability, app→org, app→process)
2. Store contextual TIME/Criticality as global on the application (most common value, or the "worst case")
3. Build a **query/view layer** that *reconstructs* the n-ary table at query time by joining across relationships:

```csharp
// Pseudo-code for reconstructing KamstrupData
var mappings = from app in container.ApplicationFactSheets
               from cap in app.GetRelatedFactSheets<BusinessCapabilityFactSheet>()
               from org in app.GetRelatedFactSheets<UserGroupFactSheet>()
               from proc in app.GetRelatedFactSheets<ProcessFactSheet>()
               select new { app, cap, org, proc, 
                           TIME = app.PortFolioStrategy.TIME,  // Global, not contextual
                           Criticality = app.BusinessSupport.Criticality };  // Global
```

### What Exists Today
- ✅ `GetRelatedFactSheets<T>()` — generic typed relationship traversal
- ✅ LINQ query capability on all collections
- ✅ `StandardGrid2` — could display the computed view

### What Needs Dev
1. View/query component that produces the cross-product (2h)
2. UI to display it (2h)
3. **Total: ~4h**

### Pros
- **Minimal dev effort** — no new data types at all
- **Uses existing model completely** — nothing new to maintain
- **Produces a familiar table view** — looks like KamstrupData

### Cons
- **Loses ALL contextual attributes** — this is the critical flaw. TIME and Criticality can only be global per-app, not per-context. This is the #1 thing Kamstrup cares about.
- **Cross-product explosion** — if App X relates to 5 capabilities and 3 orgs, the view shows 15 rows, most of which don't actually exist in KamstrupData
- **No way to distinguish real mappings from spurious cross-products** — destroys data integrity
- **Not reversible** — once you lose the n-ary context, you can't get it back

### Verdict: ❌ Not recommended. Loses the core value of Kamstrup's data model.

## Final Recommendation

### 🏆 Solution 2: ApplicationContextMapping FactSheet — with Solution 1 as Phase 1

**Implementation plan:**

#### Phase 1 (Day 1-2 of import): Solution 1 — Quick Win
- Import all binary relationships from KamstrupData
- Store contextual TIME/Criticality as auto-generated FactSheetConcerns
- **Result:** Kamstrup sees their data in OmniGaze immediately, capability maps work, app grids work
- **What's deferred:** Clean querying of contextual attributes

#### Phase 2 (Day 4-5 of import): Solution 2 — Full Model
- Create `ApplicationContextMappingFactSheet` 
- Import the 5,396 KamstrupData rows as mapping factsheets
- Wire up the contextual TIME/Criticality display in capability map views
- **Result:** Full-fidelity n-ary model, queryable, reportable

#### Why this two-phase approach:
1. Kamstrup sees value in **3 days** (Phase 1)
2. Full model fidelity in **5-7 days** (Phase 2)
3. Phase 1 work is NOT thrown away — the binary relationships created in Phase 1 are still valid and used alongside Phase 2's mapping factsheets
4. If Phase 2 turns out unnecessary (Kamstrup says "the binary view is fine"), we save the effort

#### What makes Solution 2 the right long-term choice:
- **It's the standard EA pattern** — LeanIX, Ardoq, Alfabet all model "application-in-context" as a first-class entity
- **It's reusable** — other customers with n-ary mapping needs (and they will come) benefit from the same infrastructure
- **It leverages OmniGaze's existing strength** — everything that works for FactSheets (custom fields, concerns, search, export, audit, AI surveys) works for mappings too
- **It's honest** — rather than hiding complexity in text blobs or dictionaries, it makes the data model explicit and clean
