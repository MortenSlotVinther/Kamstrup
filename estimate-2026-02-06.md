### 3.4 đź”´ Structural Challenge: The Multi-Dimensional Mapping Table

**This is the core complexity.** The `KamstrupData` sheet represents a **fact table** where each row is a unique combination of:
- Application Ă— Business Capability Ă— Organization Ă— Process Ă— Value Stream

In OmniGaze, relationships are stored as:
- Application â†’ BusinessCapability (via ParentFactSheetsIds)
- Application â†’ UserGroup (via ParentFactSheetsIds)
- Application â†’ Process (via ParentFactSheetsIds)
- BusinessCapability â†’ ValueStream (via SupportedValueStreamIds)

**The challenge:** OmniGaze relationships are **binary** (Aâ†’B), while Kamstrup's model is **n-ary** (AĂ—BĂ—CĂ—D per row with contextual attributes). A single application can map to the same capability multiple times with different org context and different TIME ratings.

**Proposed solution:**
1. **Import all binary relationships** â€“ every appâ†’capability, appâ†’org, appâ†’process relationship from KamstrupData
2. **Contextual attributes** (per-mapping TIME, criticality, lifecycle) â€“ store as `FactSheetConcern` entries on the application, with structured metadata referencing the capability/org context
3. **Future enhancement** â€“ Consider a dedicated "Mapping" or "ApplicationContext" factsheet type that explicitly models the n-ary relationship
